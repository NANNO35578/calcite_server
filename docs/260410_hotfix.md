# Hotfix: 修复 MinIO 上传异步任务悬空指针导致的段错误

**修复日期**: 2026-04-10  
**问题级别**: 严重 (Crash)  
**影响范围**: OCR 文件上传至 MinIO 功能

---

## 1. 问题描述

### 1.1 崩溃现象
程序运行出现 `SIGSEGV` 段错误，崩溃栈指向 `std::string::clear()`，实际为**悬空指针野访问**。

### 1.2 核心错误原因
- 上层 `OcrController.cc` 中调用 `MinioClient::uploadFile` 时，仅传递 `std::shared_ptr<std::vector<uint8_t>>` 的**原始裸指针 `data()`** 与 `size()`
- 函数内部将该裸指针捕获到 `queueInLoop` 异步任务与子线程中使用
- 异步线程执行时，外层 `shared_ptr` 已被释放，对应内存被销毁，裸指针变为**悬空指针**，子线程访问导致段错误
- 本质是**跨线程异步场景下，仅传递裸指针未托管内存生命周期**

### 1.3 错误代码位置
- `OcrController::processOcrWorkflow` 第 271-275 行：传参为 `fileData->data()`、`fileData->size()`
- `MinioClient::uploadFile` 原函数签名接收 `const uint8_t* data, size_t dataSize`，并在异步 lambda 中捕获该悬空指针

---

## 2. 修复方案

### 2.1 修改策略
通过修改函数签名，直接传递 `std::shared_ptr<std::vector<uint8_t>>` 智能指针，利用 `shared_ptr` 的引用计数机制延长内存生命周期，确保异步线程执行期间内存有效。

### 2.2 具体修改

#### 2.2.1 MinioClient.h - 修改函数签名
```cpp
// 修改前
void uploadFile(
    const std::string& objectKey,
    const uint8_t* data,           // 裸指针
    size_t dataSize,               // 裸指针大小
    const std::string& contentType,
    std::function<void(const UploadResult&)> callback
);

// 修改后
void uploadFile(
    const std::string& objectKey,
    std::shared_ptr<std::vector<uint8_t>> fileData,  // 智能指针托管内存
    const std::string& contentType,
    std::function<void(const UploadResult&)> callback
);
```

#### 2.2.2 MinioClient.cc - 修改实现
```cpp
// 修改前
drogon::app().getLoop()->queueInLoop([this, objectKey, data, dataSize, contentType, callback]() {
    std::vector<uint8_t> dataCopy(data, data + dataSize);  // data 可能已悬空

// 修改后
drogon::app().getLoop()->queueInLoop([this, objectKey, fileData, contentType, callback]() {
    // fileData 智能指针延长生命周期，确保内存安全
    std::vector<uint8_t> dataCopy(fileData->data(), fileData->data() + fileData->size());
```

#### 2.2.3 OcrController.cc - 修改调用处
```cpp
// 修改前
minioClient_.uploadFile(
    objectKey,
    fileData->data(),      // 传递裸指针
    fileData->size(),
    contentType,
    callback
);

// 修改后
minioClient_.uploadFile(
    objectKey,
    fileData,              // 直接传递智能指针
    contentType,
    callback
);
```

## 3. 修复验证

### 3.1 线程安全性保证
- 智能指针 `fileData` 被捕获到 `queueInLoop` lambda 中，引用计数 +1
- 子线程 lambda 再次捕获 `fileData`，引用计数 +1
- 异步任务执行期间，即使外层 `fileData` 离开作用域，内存也不会被释放
- 所有异步任务完成后，`shared_ptr` 引用计数归零，内存安全释放

### 3.2 业务逻辑保持
- 保留原有的 `dataCopy` 拷贝逻辑，避免直接在 CURL 中使用原始数据
- 所有日志输出、回调逻辑、错误处理完全保持不变
- 接口返回值、行为与原代码一致
