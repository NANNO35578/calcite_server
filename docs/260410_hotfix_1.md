# Hotfix 1: 修复 updateFileResource 中的悬垂引用导致的段错误

**修复日期**: 2026-04-10  
**问题级别**: 严重 (Crash)  
**影响范围**: OCR 工作流中的文件资源状态更新

---

## 1. 问题分析

### 1.1 GDB 调试信息分析

从 GDB 输出关键线索：
```
Thread 6 "DbLoop" received signal SIGSEGV, Segmentation fault.
0x00005555557a29e8 in drogon::orm::internal::SqlBinder::exec() ()
```

崩溃发生在：
- **线程**: "DbLoop"（Drogon 数据库事件循环线程）
- **位置**: `drogon::orm::internal::SqlBinder::exec()`
- **时间**: 在 `uploadFile` 函数返回后，新线程启动时

### 1.2 根本原因

在 `updateFileResource` 函数中存在**悬垂引用（Dangling Reference）**问题：

```cpp
void OcrController::updateFileResource(
    int64_t fileId,
    const std::string& status,
    int64_t noteId,
    const std::string& url,
    std::function<void(bool)> callback
) {
    drogon::orm::Mapper<drogon_model::calcite::FileResource> mapper(drogon::app().getDbClient("default"));
    mapper.findByPrimaryKey(
        fileId,
        [&mapper, status, noteId, url, callback](const drogon_model::calcite::FileResource& file) {
            // ...
            mapper.update(  // ❌ 使用已被销毁的 mapper！
                updatedFile,
                ...
            );
        },
        ...
    );
}
```

**问题流程**：
1. `mapper` 是 `updateFileResource` 函数栈上的局部变量
2. Lambda 按引用捕获 `[&mapper, ...]`，只保存了 `mapper` 的引用
3. `findByPrimaryKey` 是异步操作，立即返回
4. `updateFileResource` 函数返回，`mapper` 被销毁
5. 数据库查询完成后，lambda 回调被执行
6. 回调中访问 `mapper.update()` 时，`mapper` 已经是悬空引用
7. 访问已释放内存，导致 `SIGSEGV` 段错误

---

## 2. 修复方案

### 2.1 修复策略

将 lambda 中的 `mapper` 捕获方式从**引用捕获**改为**值捕获**，确保异步回调执行时 `mapper` 对象仍然有效。

### 2.2 代码修改

**文件**: `calcite/controllers/OcrController.cc`

**修改前**（第207行）：
```cpp
mapper.findByPrimaryKey(
    fileId,
    [&mapper, status, noteId, url, callback](const drogon_model::calcite::FileResource& file) {
        drogon_model::calcite::FileResource updatedFile = file;
        updatedFile.setStatus(status);
        if (noteId > 0) {
            updatedFile.setNoteId(noteId);
        }
        if (!url.empty()) {
            updatedFile.setUrl(url);
        }
        
        mapper.update(
            updatedFile,
            [callback](size_t) { callback(true); },
            [callback](const drogon::orm::DrogonDbException&) { callback(false); }
        );
    },
    [callback](const drogon::orm::DrogonDbException&) { callback(false); }
);
```

**修改后**：
```cpp
mapper.findByPrimaryKey(
    fileId,
    [mapper, status, noteId, url, callback](const drogon_model::calcite::FileResource& file) mutable {
        drogon_model::calcite::FileResource updatedFile = file;
        updatedFile.setStatus(status);
        if (noteId > 0) {
            updatedFile.setNoteId(noteId);
        }
        if (!url.empty()) {
            updatedFile.setUrl(url);
        }
        
        mapper.update(
            updatedFile,
            [callback](size_t) { callback(true); },
            [callback](const drogon::orm::DrogonDbException&) { callback(false); }
        );
    },
    [callback](const drogon::orm::DrogonDbException&) { callback(false); }
);
```

**关键变更**：
- `[&mapper, ...]` → `[mapper, ...]`：值捕获 `mapper` 对象
- 添加 `mutable` 关键字：允许在 lambda 内部修改捕获的 `mapper`（调用非 const 成员函数 `update`）

## 3. 修复验证

### 3.1 生命周期分析

修复后：
1. `mapper` 对象被值捕获到 lambda 中，引用计数 +1（或拷贝构造）
2. `updateFileResource` 函数返回，原始 `mapper` 销毁
3. Lambda 中持有独立的 `mapper` 拷贝，仍然有效
4. 数据库查询完成后，lambda 回调执行
5. `mapper.update()` 安全执行，访问的是有效的对象
6. Lambda 销毁，捕获的 `mapper` 随之销毁

### 3.2 其他潜在问题检查

检查了代码中其他类似的异步数据库操作模式：
- `createNoteFromOcr` 中的 `noteMapper`：✅ 使用的是局部变量，但没有在嵌套 lambda 中引用
- `recognize` 中的 `mapper`：✅ 直接传递，没有引用捕获问题

## 4. 为什么之前的修复没有解决这个问题

之前的修复（260410_hotfix.md）解决了 `uploadFile` 中的 `fileData` 悬空指针问题，但程序仍然崩溃，因为：

1. **第一次修复**解决了 `fileData` 智能指针生命周期问题
2. **但崩溃发生在数据库线程**，与 `uploadFile` 无关
3. **根本原因**是 `updateFileResource` 中独立的悬垂引用问题

两个问题是独立的，都需要修复：
| 问题 | 位置 | 原因 | 状态 |
|-----|------|------|------|
| fileData 悬空指针 | `MinioClient::uploadFile` | 裸指针捕获到异步线程 | ✅ 已修复 |
| mapper 悬垂引用 | `OcrController::updateFileResource` | 局部变量按引用捕获 | ✅ 本次修复 |

## 5. 建议

### 5.1 代码审查建议

1. **全面检查引用捕获模式**：
   ```bash
   grep -rn "\[&.*\]" calcite/controllers/ calcite/services/ calcite/utils/
   ```
   检查所有 `[&xxx, ...]` 模式，确认是否存在类似的悬垂引用问题。

2. **异步编程规范**：
   - 异步 lambda 中避免按引用捕获局部变量
   - 优先使用值捕获或智能指针
   - 对于 ORM Mapper 等对象，确保生命周期覆盖异步操作

3. **静态分析工具**：
   引入 `clang-tidy` 检查规则：
   ```yaml
   Checks: 
     - cppcoreguidelines-lambda-capture-reference
     - bugprone-dangling-reference
   ```

## 7. 总结

本次修复解决了 `updateFileResource` 函数中的悬垂引用问题。该问题导致异步数据库回调执行时访问已被销毁的 `mapper` 对象，引发段错误。

**修复要点**：
- 将 `[&mapper, ...]` 改为 `[mapper, ...]`
- 添加 `mutable` 关键字允许调用非 const 成员函数

这与之前的 `fileData` 修复是两个独立的问题，都需要应用才能确保程序稳定运行。
